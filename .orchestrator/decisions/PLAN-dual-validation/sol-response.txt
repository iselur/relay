The phrase “everything non-trivial” is unwise if left undefined. Applied literally to ordinary implementation work, it would consume the worker quota, slow autonomous development, and recreate the exact planning ceremony the worker policy rejects.

The defensible version is: every non-trivial control-plane decision or durable recommendation gets dual validation; routine execution inside an already-approved worker envelope does not.

## Q1. Trigger

Steelman: a broad default prevents convenient under-classification. Small-looking changes to permissions, evidence, or approval semantics can carry more risk than large code changes. Requiring two perspectives has already caught real failures, and the decision record preserves institutional reasoning across sessions.

Attack: “non-trivial” by itself is vibes. It invites inconsistent classification, task-splitting to evade review, and reviews of implementation details already covered by independent gates. It also makes Codex both architect and worker, competing for the same quota.

Use this mechanical test. A decision is non-trivial if any answer is yes:

1. Does it create or change a durable policy, invariant, architectural precedent, shared contract/schema, or mechanism intended for reuse across jobs/specs?
2. Does it change trust boundaries, permissions, credentials, isolation, security/privacy assumptions, approval/autonomy, gate behavior, evidence integrity, branch/release behavior, or recovery semantics?
3. Does it cause a destructive, irreversible, production-facing, or novel external side effect not already authorized by an established workflow?
4. Does it affect behavior beyond one approved spec’s allowed paths and acceptance/test scope?
5. Is it a research or review deliverable intended to recommend or authorize one of the above?

If none applies, it is routine. If classification is genuinely uncertain, classify it as non-trivial.

Explicit exemptions, provided none of the triggers applies:

- Read-only discovery, diagnostics, status reporting, and evidence collection.
- Clerical or mechanical changes with no semantic effect.
- Drafting a routine spec that instantiates already-decided intent and established mechanisms.
- Routine dispatch, monitoring, retry, evidence capture, and PR mechanics under existing policy.
- Worker implementation and helper work wholly bounded by an approved spec, allowed paths, test command, reviewer, and CI.
- The worker’s private implementation checklist or working plan.

Complexity, file count, and estimated duration are not triggers. A 30-file implementation can be routine; a one-line permission change is not.

## Q2. Use two tiers

Use one invariant with two review depths. A single bar either over-processes reversible design work or under-reviews security and governance.

Standard review applies to non-trivial decisions that do not meet the critical criteria:

- Claude prepares the plan and explicitly approves it for consultation.
- SOL gets one independent adversarial pass.
- SOL must return an explicit `PASS` or `BLOCK`.
- Claude disposes of every material finding.
- A `BLOCK`, or a material revision after `PASS`, requires resubmission.

Critical review applies when the decision changes policy/invariants, trust or security boundaries, approval/autonomy, gates or evidence integrity, protected branch/release behavior, shared contracts with multi-job blast radius, or has irreversible/production consequences:

- Claude and SOL iterate on the same plan revision until both explicitly report no unresolved blocker.
- Neither party’s unsupported assertion should be accepted merely for consensus; factual disagreements require evidence.
- If they cannot converge, Val decides whether to revise, reject, or explicitly waive dual validation.

The same-revision requirement matters. A SOL pass on revision A does not validate a materially changed revision B.

## Q3. Prevent plan theater

Review one coherent decision, not every file, command, or implementation step. Do not split a single decision to avoid review, and do not bundle unrelated decisions merely to amortize a consultation.

A plan earns the round-trip only if it contains:

- The decision, scope, and non-goals.
- Relevant current-state evidence and explicit assumptions.
- Plausible alternatives and why the proposed option wins.
- Affected contracts, boundaries, and downstream consumers.
- Failure modes, security implications, and blast radius.
- Ordered implementation or adoption steps.
- Validation evidence and success criteria.
- Rollback, recovery, or an explanation of irreversibility.
- Open questions and the facts that would invalidate the choice.

“N/A, because…” is acceptable. “Detailed” should mean decision-complete, not long.

A SOL consultation counts only when SOL receives the actual constraints and proposed plan and is asked to falsify it. Its response should include an explicit verdict, the strongest counterargument, questionable assumptions, important failure modes, and concrete validation gaps. A generic endorsement is not validation and should be retried.

Discovery must remain possible before planning. Read-only investigation, experiments in disposable environments, and drafting are allowed; mutation or publication of the durable recommendation waits for validation.

## Q4. Quota and latency

Do not silently degrade to Claude-only validation. That would make the invariant dishonest.

Use these controls:

- One initial SOL consultation per coherent Standard decision.
- Additional passes only after `BLOCK` or a material risk-bearing revision.
- Batch only tightly coupled decisions that genuinely share assumptions and consequences.
- Treat consultations and workers as consumers of one shared Codex budget and queue; never interrupt a running worker to launch a consultation.
- Continue unrelated routine worker and discovery work while a decision is waiting.
- Run consultations detached and never impose minute-scale timeouts.
- If SOL is unavailable or quota-exhausted, mark the decision pending and do not perform its non-trivial action.

Val may expressly authorize a scoped waiver, but it must be recorded as a waiver—not mislabeled “dual validated”—with rationale, scope, expiry, and required retrospective review. For an active incident, only the minimum reversible containment needed to stop ongoing harm should be allowed before validation.

## Q5. Exact CLAUDE.md wording

```markdown
## Architecture and governance planning invariant

This invariant governs the control/decision layer. It does not govern a worker's implementation
planning and does not add a plan-conformance gate to the worker pipeline.

Before taking a non-trivial control-plane action or publishing a durable recommendation, Claude MUST
prepare a decision-complete plan and obtain explicit validation from both Claude and Codex SOL on
the same plan revision. Read-only discovery, evidence gathering, disposable experiments, and plan
drafting may occur first.

A decision is non-trivial if it does any of the following:

1. creates or changes a durable policy, invariant, architectural precedent, shared contract/schema,
   or mechanism intended for reuse across jobs or specs;
2. changes trust boundaries, permissions, credentials, isolation, security/privacy assumptions,
   approval/autonomy, gate behavior, evidence integrity, protected branch/release behavior, or
   recovery semantics;
3. causes a destructive, irreversible, production-facing, or novel external side effect not already
   authorized by an established workflow;
4. affects behavior beyond one approved spec's allowed paths and acceptance/test scope; or
5. produces a research/review recommendation intended to decide or authorize any item above.

If none applies, the work is routine. If classification is genuinely uncertain, treat it as
non-trivial. Complexity, duration, and file count alone do not make work non-trivial.

Provided none of the triggers above applies, the following are exempt: read-only diagnostics and
status work; clerical changes with no semantic effect; routine spec drafting that instantiates
already-decided intent and established mechanisms; routine dispatch, monitoring, retry, evidence,
and PR mechanics; and worker/helper implementation wholly bounded by an approved spec, allowed
paths, test command, independent review, and CI. Worker implementation checklists and working plans
MUST NOT be cross-reviewed.

Review one coherent decision, not each command or file. Do not split a decision to evade review or
bundle unrelated decisions to amortize review.

Every plan MUST state the decision and non-goals, evidence and assumptions, alternatives and
rationale, affected boundaries, failure modes and blast radius, implementation/adoption sequence,
validation criteria, rollback/recovery, and unresolved questions. Concise "N/A, because..." entries
are permitted.

Standard decisions require one adversarial SOL pass. SOL MUST return an explicit PASS or BLOCK and
address the strongest counterargument, questionable assumptions, failure modes, and validation
gaps. Claude MUST record a disposition for every material finding. A SOL BLOCK, or a material change
to the chosen approach, scope, risk assumptions, validation, or rollback after PASS, requires
resubmission.

Critical decisions require joint iteration until Claude and SOL both explicitly report no unresolved
blocker on the same revision. A decision is Critical if it changes governing policy/invariants,
trust or security boundaries, approval/autonomy, gates or evidence integrity, protected
branch/release behavior, a shared contract with multi-job blast radius, or has irreversible or
production consequences. Unresolved disagreement goes to Val; it is not silently resolved by
Claude.

Consultations and worker runs consume one shared Codex budget. Use one initial consultation per
coherent decision, avoid duplicate or speculative rounds, run consultations detached without a
minute-scale timeout, and never interrupt a running worker for a consultation. If SOL or quota is
unavailable, the affected decision remains pending while exempt work may continue. Skipping SOL is
not compliant. Only Val may authorize a scoped, recorded waiver; emergency pre-validation action is
limited to minimum reversible containment of active harm.

Record each reviewed decision under `.orchestrator/decisions/<id>/`. The record MUST include the
trigger classification and tier, plan and revision digest, consultation prompt, complete SOL
response (and raw-stream hash where applicable), Claude's disposition, explicit Claude and SOL
verdicts, relevant model/config and commit identifiers, integrity hashes, and any waiver or
unresolved issue.

The validated decision plan is an architecture record, not a worker acceptance artifact. Any binding
requirements it creates MUST be encoded in the governing policy or approved spec before dispatch.
Workers remain judged by the approved spec, allowed scope, tests, resulting diff, independent review,
and CI—never by conformance to this plan.
```

That preserves Val’s “everything non-trivial” decision in a usable form. If he instead intends every ordinary multi-step worker change to receive a separate SOL-reviewed plan, I would advise against it plainly: that would contradict the existing worker architecture, concentrate planning and execution in the same reviewer, and spend the constrained execution quota on a redundant gate.
