=== MY DRAFT DESIGN (challenge every part) ===
1. Create system user `codex-worker` (no login shell, no sudo) + shared group `codexwork` with
   members {val, codex-worker}. Only the `codex exec` subprocess drops to codex-worker (via `sudo -u
   codex-worker` or a `systemd-run --uid`); the rest of `_run` stays val.
2. Worktrees move OUT of /home/val to a setgid shared dir, e.g. `/srv/codexwork/worktrees`
   (`val:codexwork`, mode 2770). val creates the worktree + later reads worker output to commit;
   codex-worker (in codexwork) edits files + runs the test_command there. The worker never runs git
   (G1-A/C), so it doesn't need the parent .git (which stays unreadable in /home/val).
3. Codex reachable by codex-worker: COPY the 336 MB codex install to a shared path `/opt/codex`
   (0755), and re-point the AppArmor `userns` profile at `/opt/codex/.../bwrap`. codex-worker runs
   `/opt/codex/bin/codex`.
4. Codex AUTH for codex-worker: copy `val`'s `~/.codex/auth.json` into
   `/home/codex-worker/.codex/auth.json` (600 codex-worker). This SHARES the Codex subscription
   identity but keeps it unreadable to... itself is fine (it's the worker's own creds); the point is
   codex-worker still cannot read gh/ssh/claude. Alternative: a SEPARATE `codex login` for
   codex-worker (needs Val's device-auth; possible second-seat concern).
5. KEEP bwrap as defense-in-depth (write+network confinement) on top of the user boundary.
